[README](../README.zh-TW.md) 勾勒了整個概念,本文件補上背後的設計細節。先從那裡讀起,掌握定位與詞彙;

## 三種畫面,一座座艙

Baton 是純鍵盤操作的,提供三種觀看 agent 的方式。儀表板與放大是兩個極端——一次看全部,或把一個看透——
而群組分割坐落於兩者之間,把一個 work item(工作項目)的成員即時並排攤開來。

- **儀表板(Dashboard)** — 一次看到全部。你在這裡導覽面板、開新的、把它們併成工作項目、重新排序
  (`shift`+方向鍵),並退掉已經死掉的。
- **群組(Group)** — 即時觀看一個工作項目。放大一張工作項目卡片,會開出它成員的分割畫面,每個成員各佔一磚,
  全部同時串流(見[工作項目](#工作項目work-items))。這是你用來導覽的總覽,而不是可以打字進去的畫面——除非你
  按下 `i`。
- **放大(Zoom)** — 把一個東西看透。像操作你唯一的終端機一樣驅動單一面板;`C-t [` 開啟 tmux 風格的捲動模式
  (`↑`/`↓` 一行,`b`/`space` 或 `PgUp`/`PgDn` 一頁,`g`/`G` 到頂/到底,`esc` 離開),讓你回頭讀它的歷史。
  `C-t` 領導鍵在你捲動時仍然有效——每個逃脫(`C-t d` 到儀表板、`C-t f` 搜尋捲動歷史、`C-t /`、`C-t ~`)都不必先
  離開捲動模式就能觸發——而一個面板會記住你捲到哪裡,所以重新放大它會回到同一個位置,而且仍在捲動模式。
  每個裸按鍵都驅動程式(vim、BBS),永遠不會驅動 baton,所以領導鍵在任何終端機上都能用——然後再彈回儀表板。
  在接上時保留與重播多少歷史,由設定中的 `panel.replay-kb` 決定(值越大能往回翻越多;全螢幕程式不保留捲動歷史)。

你永遠不用在視窗或分頁間手忙腳亂。你從儀表板指揮,當某項任務需要細看時就落進一個群組,只有在某個演奏者需要你時
才放大進去。

## 面板(Panels)

一個面板就是伺服器所擁有的一個 PTY(pseudo-terminal,虛擬終端機)。它有兩種:

- **Agent 面板** — 直接把 agent CLI 當成面板的行程來跑。中間沒有 shell、沒有 shell 提示字元;agent CLI
  _就是_ 這個 PTY 所跑的程式。
- **Shell 面板** — 跑一個純粹的宿主 shell,用來在機器上臨時下指令。

兩者都是普通的 PTY,共用底下的生命週期;它們只差在啟動的是哪個行程,以及 Monitor 為了引起你的注意會把它們喊得
多大聲。

**Agent 設定檔(profiles)。** 一個 agent 面板是從一份具名的 **profile**(一個指令與它的引數)產生的,並在你選定的
**工作目錄**(agent 操作的目錄)裡執行。**Claude** 是內建的 profile(`claude`);更多的則定義在設定中的
`panel.agents` 下,而 `panel.default-agent` 指名新增 agent 這個動作要產生的是哪一個。客戶端解析 profile 後送出
`panel.create`,帶著指令、引數與 workdir;伺服器在那裡啟動行程,而面板的標題會顯示為 `<command> · <workdir>`
(例如 `claude · baton`),讓它的任務與所在位置一眼可見。

### 生命週期

Monitor 帶著面板走過一小組狀態,從你產生它的那一刻,到你關掉它的那一刻:

```txt
                  ● spawn
                  │
                  ▼
             ┌──────────┐
             │ spawning │
             └────┬─────┘
                  │ process started
                  ▼
     ┌──────────┐                           ┌──────────┐
     │ running  │  ──── output quiet ────>  │   idle   │
     │          │  <─── output resumes ───  │          │
     └──┬────▲──┘                           └────┬─────┘
        │    │                                   │
  needs │    │ replies                           │
  input ▼    │                      needs input  │
     ┌──────────┐                                │
     │ attention│ <──────────────────────────────┘
     └────┬─────┘
          │ process exits      (also from running / idle)
          ▼
     ┌──────────┐
     │  exited  │
     └────┬─────┘
          │ user dismisses
          ▼
     ┌──────────┐
     │  closed  │ <──── kill panel (from any state)
     └──────────┘
```

- **spawning** — PTY 與行程正在建立中。
- **running** — 行程正積極地產出輸出。
- **idle** — 有一陣子沒有輸出;某個 agent 正在等候,或某個 shell 停在它的提示字元。
- **attention** — 需要你了:agent 問了問題、完成了任務,或印出了某些值得注意的東西。
- **exited** — 行程自己結束了;它的離開碼會保留到你把它撤掉為止。
- **closed** — 面板已退役,並離開儀表板。

`running`、`idle` 與 `attention` 是幾個即時狀態,Monitor 會隨著輸出的起落把面板在它們之間來回擺渡。一個面板可以
**從任何狀態被 kill**,直接跳到 `closed`;而只要它的行程自己停下——不論是從 `running`、`idle` 還是 `attention`——
它就會到達 `exited`。

## 工作項目(Work items)

一個 **work item(工作項目)** 是一組具名、同屬一項任務的面板。群組成員關係就只是每個面板上帶著的一個名字
(`group`),所以一個工作項目 _就是_ 它的名字:把面板歸檔到 `"api"` 底下就讓它們成為同一個項目,重新命名群組就把
那個名字改寫到每一個成員身上,而移除一個面板只會清掉它自己的名字。伺服器擁有這一切——分組、移除與重新命名都是核心
動作(`panel.group`、`panel.ungroup`、`panel.rename`),透過 socket 觸達——所以每個前端、每次重新接上看到的群組都
一致。`panel.ungroup` 接受一個群組名稱(解散整個項目)或一組 panel id(只丟掉那些成員)。

**名稱預設是唯一的。** 伺服器會拒絕名稱已屬於另一個面板標題或群組的重新命名或新群組,所以一個工作項目永遠不會有
歧義。把面板加進一個 _既有_ 群組會重用它的名字,這是允許的;這項政策可以用 `allow-name-conflict` 設定解除,常駐
程式會在啟動時讀取它。

**巢狀群組。** 群組名稱是一條 **斜線路徑(slash-path)**,所以一個群組可以巢狀在另一個裡面:歸檔到
`"backend/api"` 底下的面板坐落在群組 `api`,巢狀在 `backend` 之下。成員關係仍是推導出來的——只要有任何面板帶著
`backend` 或它底下的一條路徑,`backend`「就存在」。要巢狀,你可以在分組時用一條路徑(`backend/api`)、
把**一個群組重新命名成一條路徑**——把 `db` 改名為 `backend/db` 會一次把整個 `db` 子樹重新掛到新的父層(伺服器會
把路徑前綴改寫到每一個後代上)——或是**把整個群組併入或加入另一個群組**:標記一個群組,再把它併入或加入一個目標,
它就會巢狀成 `target/<它的名字>`(保留它自己的子結構),而不是被攤平。整組範圍的動作會**沿子樹遞迴**:對
`backend` 派工、關閉或送訊號,會觸達每一個後代面板,巢狀群組也包含在內;**解散** `backend` 會把它的子樹往上提升
一層(它的直屬面板變成獨立面板,它的子群組升為頂層),而不是刪掉那些工作。儀表板只顯示**頂層**——一張群組卡片會
把它整個子樹摺起來——而你透過在分割畫面裡下鑽來走訪這個階層。

在儀表板上,一個工作項目會塌縮成單一一張卡片:一個成員計數,以及一個**上捲到它最緊急成員**的狀態
(attention 勝過 running 勝過 spawning 勝過 idle 勝過 exited),所以一張卡片就替整項任務發聲。

**最愛(Favourites)。** `*` 會把選中的儀表板項目**加入最愛**——不論是一個獨立面板還是一整張群組卡片——而被加入
最愛的卡片會排序到儀表板的**最前面**,不論在網格還是樹狀中,各以一個 `⊙` 標示。這是**伺服器所擁有的狀態**
(面板用 `panel.favourite` / `panel.unfavourite`,群組用 `group.favourite` / `group.unfavourite`),帶在快照上——
所以它能撐過重啟、在各客戶端間共享,並且跟著群組經歷重新命名或解除群組。它與分割畫面的**釘選是分開的**:加入最愛
只重新排序儀表板,絕不影響哪些磚即時串流,也不影響單一釘選的下鑽。

**群組分割。** 放大一個工作項目會開出一個範圍限定在那條路徑**直屬子項**的分割畫面——每個直屬面板在它自己的磚裡
即時算繪,外加每個直屬**子群組**各佔一磚(一個以 `▣` 標示的上捲方塊)。分割畫面預設是一個 _你用來導覽的總覽_,
而不是可以打字進去的表面:`tab` 在磚之間移動焦點,`+`/`-` 調整有多少面板即時串流(見下文),`shift`+`←`/`→` 在
群組內重新排序聚焦的成員,`C-t [` 對聚焦的磚開啟捲動模式,`x` 把聚焦的成員從群組移除,`D` 對聚焦的 agent 成員做
diff,而 `L`/`z` 輪替版面配置 / 調整一個磚的大小(view-local,見 [TUI.md](./TUI.zh-TW.md#縮放))。`enter` 會
**下鑽**:在一個面板上,它會落進那個面板自己的單一放大;在一個子群組磚上,它會把分割畫面的範圍重新限定到那個
子群組(它的標頭會把路徑顯示成一條麵包屑,`backend › api`)。`esc` / `b` 會**退回上一層**——子群組退回父層、摘要
子檢視退回它的群組、頂層群組退回儀表板——而 `d` 會從任何深度直接跳出到儀表板。從一個放大中的成員,**back**
(`C-t b`)會退回它當初是從哪個分割畫面啟動的。你用卡片上的 `enter` 進入一個群組,再用 back 走出來。

**可見數量與摘要磚。** 一個群組會把它的前 **N** 個成員當成即時磚串流;`+` / `-` 調整 N,這是**伺服器所擁有的
狀態**(`group.show`,以群組的 `Shown` 帶在快照上),被夾在 `[1, maxGroupTiles]` 之間並持久化,所以分割畫面會照你
離開時的樣子重新開啟。為了不讓其餘成員被晾在一旁,每個超過 N 的成員都會收摺進單一一個**摘要磚**——均勻網格的最後
一格——它把被隱藏的成員上捲起來:它們的數量、逐狀態的細分,以及最緊急的活動列。`tab` 會走過即時磚,然後把摘要格當
成同一個環的一部分走過去。聚焦摘要並按 `enter` 會把它放大成**只有被摺起成員的子網格**;`esc` 會回到父群組,而不是
儀表板,而釘選 / 互動 / 訊號 / 移除等按鍵在摘要上會無作用,並給一句提示。

**釘選,給擁擠的群組用。** `p` 會**釘選**聚焦的成員:一個被釘選的面板永遠佔著一個即時磚(以 `⊙` 標示),把它提升
到自動填入的磚之前。所以你能策劃一個忙碌群組裡的哪些面板即時串流、哪些收摺進摘要。釘選是**伺服器所擁有的狀態**
(`panel.pin` / `panel.unpin`),帶在伺服器所廣播的面板上——所以它能撐過前端重啟,並在各客戶端間共享。釘選集合會在
每次快照時,從父群組的完整成員關係重新推導,所以在摘要子檢視開著時抵達的一次刷新,不會抹掉你的策劃。重新開啟一個
群組會把你釘選的磚帶回來;而一個恰好只有**一個**釘選成員的群組會把它當成預設——進入該群組會直接落進那個面板的放大,
而不是一個只有一磚的分割(**back**,`C-t b`,會退回分割畫面)。

**訊號(Signals)。** `s` 會開啟一個常見訊號的挑選器(或按 `o` 輸入任何名稱或數字);選中的那個會透過 socket 送出
(`panel.signal`),並送達該面板的整個行程群組,所以它會觸達前景工作,而不只是 shell——不過一個把自己 daemon 化
成獨立群組的子行程會逃離它,而且送達是「射後不理」的(一個被攔截的訊號仍算是已送出)。目標跟著檢視走:儀表板上是
選中項,分割畫面裡是聚焦成員(`s`)或每一個成員(`S`),放大裡是這個面板(`C-t s`)。已結束的面板在兩邊都會被
跳過,所以回報的計數就是實際送達的計數。名稱→訊號的對照表放在同一個地方(`internal/signals`),由挑選器與伺服器
共用,所以選單與被接受的集合不可能漂移。**重載。** `C-t R`(或對常駐程式送一個 `SIGHUP`)會就地重讀設定——名稱
政策、預設 workdir 與重播緩衝區會在整隊執行中的情況下改變,不必重啟,也不會弄丟任何面板。

**Diff。** `D`(`diff` 綁定,在放大裡是 `C-t D`)會顯示聚焦 **agent** 面板的工作樹 diff。它按設計就是只針對
agent——一個 shell、一張群組卡片,或一個空的選擇都無法解析出目標,客戶端只會提示 `diff: select an agent panel`,
所以這個動作在沒有面板的情況下永遠不會抵達伺服器。伺服器會以面板的 workdir 為關卡:它必須位在一個 git 工作樹裡,
否則請求會以 `not a git repository: <dir>` 失敗;一個沒有東西可顯示的樹會以 `no uncommitted changes` 失敗。變更
檢查(`internal/gitdiff`)會讀取 `git status --porcelain`,它的輸出**也會列出未追蹤檔**,所以一個 agent 剛寫出的
全新檔案也算數——而 diff 本身是工作樹對 `HEAD` 的比較,**不會變動 repo**:不寫入索引、工作樹或任何 ref——不過
把東西暫存進那個用完即丟的索引確實會把 loose blob 寫進 `.git/objects`,那些是無害的,之後一次 `git gc` 就會回收。
diff **指令**會依優先序解析:一個明確的 `panel.diff-command` 勝出,透過 `sh -c` 執行,讓使用者能寫一整行 shell
或用管線;否則,若 repo 設定了 `diff.tool`,就以 `git difftool -d --no-prompt` 尊重那個 per-repo 的選擇
(`--no-prompt` 讓 difftool 不會卡在 PTY 裡的 `[Y/n]` 上,而這條分支保留 git 只針對已追蹤檔的 difftool 語意——
納入未追蹤檔是明確路徑與內建路徑才有的性質);否則就是一個內建、不變動、納入未追蹤檔的 `git diff`,它會自然地
分頁。difftool 分支的一個注意事項:若 repo 的 `diff.tool` 是一個 GUI 工具,而常駐程式跑在無頭環境或透過 SSH 且
沒有顯示器,`git difftool` 會出錯或卡住——這個失敗會被侷限在 diff 面板裡,不是致命的,但在無頭 / 遠端主機上,
偏好用終端機 diff(讓 `diff.tool` 留空以用內建的,或把 `panel.diff-command` 指向一個終端機工具,例如
`git diff | delta`)。diff 會透過 git 的 pager 分頁,並且可導覽;但若 repo 停用了 pager(`core.pager=cat`),或
明確的 `panel.diff-command` 不分頁,一個非常大的 diff 會整個倒進面板,而只有最後的 `panel.replay-kb` 會留在捲動
歷史裡——頂端會被截掉。

diff 會算繪在一個**短暫、用完即棄的面板**裡。伺服器會把它當成一個 PTY 產生,在 agent 的 workdir 裡跑那條解析出
的指令,但它永遠不會被插進 `s.panels`——所以它對儀表板網格與狀態檔都不可見,不會被當成整隊成員廣播,也不會在重啟
時被還原。它是**自動放大**的:客戶端會直接落進它,成為一個單一放大,重用每一項閱讀便利設施——捲動模式(`C-t [`)、
捲動歷史搜尋(`C-t f`)與複製。它的生命週期綁在那個放大上:一般的離開按鍵會關掉它。`C-t d`(回到儀表板)與
`C-t q`(卸離)會撤掉這個彈出視窗,伺服器則拆掉那個短暫面板——沒有另外的「關閉 diff」動詞,而你一旦踏出去,就不會
有任何東西殘留。一個連線最多同時只能有少量開著的 diff 彈出視窗(目前是 8 個);超過之後,diff 按鍵會回報
`too many open diffs (max 8) — close one first`。

**Git 選單。** 在放大裡按 `C-t g` 會替被放大的 agent 開啟 **git 選單**——一個按鍵驅動的彈出視窗(訊號挑選器的
形狀),裡面是針對那個 agent 的 workdir 執行的 git 操作。它是**只限放大的**(你只對你正在看的那一個 agent 動手)
且只限 agent。那些**非互動式輸出**的操作(log、status、stage、push、branch、worktree-list)由 `gitops.Capture`
一次性執行,並以一個 `gitout` 訊息回覆,座艙會把它顯示在一個**可捲動的文字彈出視窗**(`modeGitOut`)裡——它是
diff 彈出視窗的文字手足:沒有 PTY,`s.panels` 或狀態檔裡都沒有東西,`esc` 關閉它。一個非零離開碼仍會開啟彈出
視窗(標頭染色),好讓 git 自己的訊息顯示出來;這些捕捉會在 `GIT_TERMINAL_PROMPT=0` 下、以 30 秒為上限執行,
所以一個會索取憑證的 push 會快速失敗,而不是卡住。只有 **commit** 保留那個短暫、自動放大的 PTY 面板(透過那個
明確的 `diff-command` 共用的 `openEphemeral` 引擎,8 個面板的上限):它會把設定好的編輯器注入成 `GIT_EDITOR`
(透過 `ptymgr.Spec.Env`),讓它在面板的 PTY 裡開啟。另外兩個操作兩者皆非:**worktree-add** 會在一個新分支上建立
一個樹,並產生一個以它為根的 agent——歸檔在該分支底下,以一次真正的整隊變更廣播——這就是面板模型 roadmap 所命名的
**隔離橋(isolation bridge)**;**worktree-remove** 會同步執行並以一則通知確認。這組操作是純增添的(沒有
reset / clean / discard / `--force`);`push` 與 `worktree-remove` 會先確認。線路上是單一一個指令 `panel.git`,
帶著操作、目標 id,以及一個分支或 worktree 路徑;只限 agent 與工作樹這兩道關卡在伺服器上具權威性。commit 編輯器與
worktree 基準目錄是 `panel.editor` / `panel.worktree-dir`,和其餘設定一樣可熱重載。完整的操作表與設定見
[GIT.md](./GIT.zh-TW.md)。

**持久化與重生。** 常駐程式能撐過自己的重啟。在每次結構性變更時,它會把整隊寫進一個 per-session 的**狀態檔**
(`internal/state`,像 pid 檔一樣從 socket 路徑推導出來,每個 session 一個常駐程式)——每個面板不可變的產生規格
(指令、引數、workdir)、群組成員關係、釘選、順序、id 計數器,以及每個群組的可見磚數。寫入會透過一個一格深的髒
通道匯整,並由一個 saver goroutine 在熱鎖之外沖刷;`SIGINT`/`SIGTERM` 的處理器會在離開前呼叫 `SaveNow`,所以即使
`os.Exit` 跳過了 saver,最後一個動作也能撐過一次乾淨的關機。存檔是原子且耐久的(暫存檔、fsync、rename、fsync
父目錄);一次載入永遠不會讓開機硬性失敗——缺檔是一次乾淨的初次執行,而一個無法解析或 schema 較新的檔案會被改名
挪到一旁(`.corrupt-<ts>`),而不是把常駐程式卡死。還原刻意是**惰性的**:每個面板都以一個已結束的死亡空位回來,
沒有任何行程自動重生(不論 shell 或 agent),而 id 計數器會從還原到的最高 id 之後續接,所以一個新面板永遠不可能
撞號。`panel.respawn` 動作(儀表板的 `r` 鍵)會依保留的規格,按需重跑一個已結束的空位——每個死亡空位一個指令,所以
對一個聚焦群組按 `r` 會一次重啟每一個已結束的成員,而在群組分割裡按 `r` 則重跑聚焦的磚;關閉或清除一個面板會把它的
規格永久丟掉。

**互動模式(Interact mode)。** 按 `i` 會把鍵盤交給聚焦的磚,讓你 _就地_ 驅動它的程式,不必進入全螢幕的放大——
那個磚會發出綠光並戴上一個鍵盤徽章,而每一個鍵擊都會被轉發給那個面板。像放大一樣,前綴是唯一的出路:`C-t i` 回到
導覽,`C-t d` 離開去儀表板,`C-t q` 卸離,而 `C-t C-t` 送出一個字面的前綴。只有聚焦的磚會收到輸入;其餘的維持被動,
所以在你選擇加入之前,導覽鍵永遠不會和某個面板可能想要的東西產生歧義。若正在被打字的那個面板離開了群組,互動會結束,
而不是無聲地重新瞄準焦點落到的那個磚。

在底層,單一一個客戶端會同時接上每一個成員;伺服器會替每則輸出訊息標上它的 panel id,客戶端則把它解多工到對應的
磚裡,同時每個磚的輸入側都會被轉發,好讓互動能觸達 PTY。分割畫面會在每次快照時協調——在別處被加入或移除的成員會
就地出現與消失,一個被清空的群組會離開回儀表板,而即時磚有上限,所以一個非常大的群組不會產生無上限的終端機。

## 任務與佇列

一個**面板**是一個終端機;一個**任務(task)**是你交給某個面板的一份工作單位。打字進一個面板是原始鍵擊——伺服器
看到的是位元組。一次**派工(dispatch)**則不同:你把 _目標_ 交給伺服器,它會把目標記在面板上,並當成一個整體送達
給行程。這份簡報接著會顯示在面板的卡片上(一列 `▸`)以及快照裡,所以每個前端、每次重新接上都知道每個 agent 正在
做什麼。

**派工(Dispatch)** 是直接路徑。儀表板上的 `T`(放大裡是 `C-t T`)會開啟一行提示;你輸入的內容會送達給聚焦的
agent——或者,在一張工作項目卡片上,**散發給群組的每一個成員**,這就是讓 N 個 agent 賽跑同一個提示背後的機制。
conductor 不能派工給它自己的面板(見 [CONTROL.md](./CONTROL.zh-TW.md))。送達會**等待就緒**:送給一個忙碌 agent
的簡報會被扣住,並在面板安定到 `idle` 或 `attention` 的那一刻才寫入,所以它會落在一個提示字元上,而不會和一個執行
中的指令交錯。

**一個任務是一個被追蹤的實體。** 每次派工都會變成一個任務,帶著一個只能前進的狀態——`queued → dispatched →
running → done`(或從任何非終端狀態進入 `failed`)——一個嘗試計數,以及它所屬的面板與群組。Monitor 會隨著輸出
流動推進它:送達把它移到 `dispatched`,第一則輸出移到 `running`,一次安定移到 `done`,一次行程離開或一個被關閉的
面板移到 `failed`。一個失敗的任務會保留一句簡短的**原因**(`panel exited (code N)`、`panel closed`);完成的任務
會以歷史逗留,上限為最近的 50 筆,所以一場很長的 session 永遠不會無上限地成長。

**佇列(The queue)** 是給還沒有地方跑的工作用的。與其派工給一個具名面板,你可以**入列(enqueue)**一份簡報
(可帶一個選擇性的群組)——在座艙裡用 `t`(`T` 派工的日常手足),或透過 `baton ctl` / MCP——然後一個伺服器所擁有
的排程器會在稍後的一個 tick 上,把它抽取到一個空閒的 agent 上,並遵守 per-group 的並行上限。任務會**依最高優先序、
再依最舊**抽取;一個等候中的任務可以被重新排到待辦佇列的頭或尾。這就是旗艦級的 **你 → conductor → 隊伍** 流程:
你把一批簡報交給 conductor,它把它們入列,然後它們流向任何空出來的工作者。

**按需產生(Spawn-on-demand)。** 一個入列的任務可以帶著一份產生規格——一個指令、引數、workdir,以及一個
close-on-done 旗標。當排程器找不到空閒 agent 時,它會替這樣的任務**佈建一個全新的 agent**(在 64 個面板的整隊
上限之下),把任務派工到那裡,並在被要求時,於任務安定後回收那個 agent。待辦佇列於是變成一個真正的工作佇列,會長出
它自己的短暫工作者,而不只是一個蓋在常備隊伍上的緩衝區。一個沒有產生規格的任務,仍然只搭乘整隊裡已有的 agent。

待辦佇列是**持久化的**。每個任務是 `$HOME/.baton/<session>.queue/` 底下的一個 JSON 檔,原子地寫入,並在任務離開
待辦佇列時由排程器移除——所以一次常駐程式重啟會還原佇列。一個已經指派給某個沒撐過重啟的面板的任務會被重新入列
(保留它的 id 與嘗試計數),而不是遺失。佇列是**選擇性加入的(opt-in)**:沒有 `queue` 設定區塊時,常駐程式會接受
派工但不跑排程器,所以自動抽取永遠不會嚇到你。`queue.max` 為未指派的待辦佇列設上限;`queue.concurrency` 為一個
群組同時可跑多少任務設上限。

**管理待辦佇列。** `Q`(放大裡是 `C-t Q`)會開啟佇列管理員——每個任務一列,顯示它的狀態、id、群組與簡報(一個
按需產生的任務會標上 ⚡,一個完成的則顯示它的原因)。`K` / `J` 把高亮的**入列中**任務升 / 降到待辦佇列的頭 / 尾,
`d` 取消它(一個已在某面板上飛行中的任務會被留著跑完;用關閉或送訊號給它的面板來取消它),而 `D` 抽乾整個未指派的
待辦佇列。這個彈出視窗不擁有自己的任何狀態:每次變動都是一個伺服器動作,而它回覆的那份全新快照——依執行順序排列的
即時待辦佇列,底下是完成的歷史——就是用來重畫這份清單的東西。

一個外掛可以塑形流過的任務——對每次狀態變更做出反應,或在一份簡報被送達之前改寫 / 否決它。見 PLUGIN.md 裡的
[task hooks](./PLUGIN.zh-TW.md#任務與佇列)。

## 尋找、搜尋、複製

**尋找(Find)**(`f`,在儀表板上)會過濾整隊:一邊打字,就只有標題或群組——或某個群組成員的標題——符合的面板留
下來,標頭會顯示符合的計數,`enter` 保留過濾而 `esc` 清除它。**搜尋(Search)**(`C-t f`,在放大裡或在一個聚焦
群組磚上)會對捲動歷史跑一個不分大小寫的正規表示式:檢視會跳到最新的命中,並停在捲動模式,把命中高亮,`n` / `N`
走訪較舊 / 較新的命中,而一個不是有效正規表示式的詞會被當成字面比對。**整隊搜尋(Fleet search)**(`/`,在儀表板
上;放大裡是 `C-t /`)會一次 grep _每一個_ 面板:伺服器會掃描每個面板保留的輸出以尋找該詞,並回傳符合的行,彈出
視窗會把它們依面板分組列出,並把該詞高亮。`j` / `k`(或 `n` / `N`)走訪命中,而 `enter` 放大命中所在的面板,在那裡
把該詞重新當成一次捲動歷史搜尋跑一遍,讓放大直接開在命中處——整隊 grep 直接交棒給那個確切的 per-panel 搜尋。它讀取
的是捲動歷史所顯示的那份有上限的重播緩衝區,所以它搜尋的是近期輸出,而不是全部歷史,而同樣的字面 fallback 規則也
適用。**複製(Copy)** 存在於捲動模式(`C-t [`)裡:`v` 開始一次選取,`y` 把選中的行——或者在沒有選取時,把可見的
那一頁——透過 OSC52 複製到系統剪貼簿,所以它在 SSH 上也能用,不需要輔助執行檔。`V` 則改為開始一次**區塊**(矩形)
選取——同樣的那些列,但只有欄 `[0, n]`,用 `h`/`l` 把右緣拉進拉出——所以你能從對齊的輸出裡,把一個窄窄的左欄抬出來。

## 滑鼠(Mouse)

滑鼠**預設是關閉的**,好讓你終端機自己的選取與複製仍可用。在按鍵對應裡切換它(`C-t k`,那個 settings 區塊);
一旦打開,滾輪會在一個放大或磚裡捲動捲動歷史,並移動儀表板的選擇,而**在群組分割裡左鍵點一下會聚焦指標下的那個磚**
(所以你能直接跳到某個成員,而不必用 tab 移過去)。這個切換會持久化在設定裡(`settings.mouse`)。

## 設定(Settings)

有幾項行為是持久化的座艙切換,在按鍵對應的 settings 區塊(`C-t k`)裡編輯,並寫進設定:

- **confirm-on-close**(預設開啟)— 用 `w` 關閉單一面板會先問 `y/n`。關閉一整個**群組**永遠會確認,並說明它會退掉
  多少個面板,不論這個切換為何——一個工作項目絕不會在一個鍵擊裡消失。
- **allow-name-conflict**(預設關閉)— 解除唯一名稱政策,讓兩個工作項目可以共用一個名字。
- **bell**(預設開啟)— 當一個面板進入 `attention` 時響鈴。
- **mouse**(預設關閉)— 見上文。

這張圖,由上往下讀:

| 區塊                 | 角色                                                        |
| -------------------- | ----------------------------------------------------------- |
| **Frontends**        | 無狀態的客戶端——算繪事件、送出指令。TUI、瀏覽器、…          |
| **The socket**       | 那唯一的一條線——語意化、有版本、經協商。指令往上,事件往下。 |
| **baton server**     | 掌管所有狀態與終端機的背景常駐程式;比任何前端活得更久。     |
| **Connection layer** | 多客戶端接上 / 卸離 / 廣播;指令 → 核心動作。                |
| **baton.\* API**     | 唯一的入口關卡——socket、Lua 與事件全都經由它通過。          |
| **Lua runtime**      | 以 Lua 寫成的設定、hook 與指令,全都呼叫 `baton.*`。         |
| **Core actions**     | 唯一真理來源;每個請求都落在這裡,而且只落在這裡。            |
| **State**            | 掌管面板、工作項目、任務與版面配置;佇列會做成快照存到磁碟。 |
| **PTY manager**      | 產生並餵養每個面板背後的真實行程。                          |
| **Monitor**          | 監看面板的存活、閒置與值得注意的輸出。                      |
| **Event dispatcher** | 把每次變更廣播給訂閱者與 hook。                             |
| **Panels**           | 那些即時終端機本身——每個都是一個 agent 或一個 shell。       |
